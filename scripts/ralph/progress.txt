## Codebase Patterns
- Use `@openrouter/ai-sdk-provider` (not `@ai-sdk/openai`) for OpenRouter integration with Vercel AI SDK
- Vercel AI SDK v5+ uses `stopWhen: stepCountIs(N)` instead of deprecated `maxSteps`
- Tool definitions use `tool()` with Zod `inputSchema` and `execute` function
- `generateText` returns `{ text, steps, toolCalls, toolResults, usage, totalUsage, response }`
- `onStepFinish` callback available for progress tracking during tool loops
- OpenRouter provider: `createOpenRouter({ apiKey })` then `provider('model/name')`
- Existing tools have executor functions that can be reused — just wrap with Zod schemas
- DataStore pattern (Map<string, unknown>) passes through tool context — reusable in SDK format
- All AI constants live in `src/lib/constants/ai.ts` — update MAX_TOOL_ROUNDS references
- Schema context is static (generateSchemaContext), user context is async (buildUserContext)
- Report progress tracking uses fire-and-forget DB updates via updateProgress()
- Chat tool loop: MAX_TOOL_ROUNDS_CHAT = 10, Report: MAX_TOOL_ROUNDS_REPORT = 20
- System prompt built by buildSystemPrompt() in src/lib/ai/prompts/trading-analyst.ts
- Tool executor pattern: executeTool() switch-case dispatches to individual executor functions
- Report email sent via Resend after completion — unchanged in migration

### Migration Mapping
| Current | Vercel AI SDK |
|---------|---------------|
| `chatCompletion(opts)` | `generateText({ model, system, messages, tools, stopWhen })` |
| `chatCompletionStream(opts)` | `streamText({ model, system, messages, tools })` |
| `for (round < MAX_ROUNDS)` | `stopWhen: stepCountIs(MAX_ROUNDS)` |
| `ToolDefinition` JSON Schema | `tool({ description, inputSchema: z.object(...), execute })` |
| `executeTool()` switch-case | Individual `execute` functions in tool definitions |
| `response.usage.total_tokens` | `result.totalUsage.totalTokens` |
| `response.choices[0].message` | `result.text` / `result.toolCalls` |
| Manual tool result messages | Automatic — SDK handles tool result → model loop |
| `buildHeaders()` | `createOpenRouter({ apiKey, headers })` |
| `fetchWithRetry()` | SDK built-in retry via `maxRetries` param |
| `MAX_TOOL_ROUNDS_CHAT = 10` | `stopWhen: stepCountIs(10)` |
| `MAX_TOOL_ROUNDS_REPORT = 20` | `stopWhen: stepCountIs(20)` |

### Vercel AI SDK Key Patterns
1. **Provider setup:** `createOpenRouter({ apiKey: env.OPENROUTER_API_KEY })` from `@openrouter/ai-sdk-provider`
2. **Model selection:** `provider('moonshotai/kimi-k2')` returns a model instance
3. **Tool definition:** `tool({ description: '...', inputSchema: z.object({...}), execute: async (args) => {...} })`
4. **Multi-step:** `generateText({ ..., tools, stopWhen: stepCountIs(10) })` — SDK handles the loop
5. **Progress tracking:** `onStepFinish({ text, toolCalls, toolResults, usage, stepType })` callback
6. **Token tracking:** `result.totalUsage` aggregates across all steps
7. **Steps access:** `result.steps` array contains all intermediate tool calls/results
8. **System prompt:** Separate `system` param (not first message)

### Package Decision
- PRD says `bun add ai @ai-sdk/openai` — but we need `@openrouter/ai-sdk-provider` for OpenRouter
- Correct install: `bun add ai @openrouter/ai-sdk-provider`
- The `@ai-sdk/openai` package targets OpenAI directly, NOT OpenRouter

## 2026-02-15 - US-009
- Created `src/lib/ai/report-pipeline/validator.ts` — the fourth and final phase of the 4-phase report pipeline
- Three validation checks:
  1. **Component validation**: Parses all `<PascalCase ...>` tags in MDX, checks against `KNOWN_COMPONENTS` set (mirrors `mdxComponents` in `src/components/mdx/components.tsx`)
  2. **DataRef validation**: Checks that dataRef-requiring components (charts, display) have a `dataRef` attribute and that its value exists in `dataStoreKeys`
  3. **MDX compilation check**: Uses `compileMDX` from `next-mdx-remote/rsc` with `sanitizeMdxProse` preprocessing and `remarkGfm` — same pipeline as the report viewer page
- Auto-repair: up to 2 attempts using `aiGenerateText` with a repair-specific system prompt listing specific errors and available dataRef keys
- Graceful degradation: if repair fails after 2 attempts, returns original content with error list (report still usable via markdown fallback)
- Returns `{ valid: boolean, content: string, errors: string[], tokensUsed: number }`
- Files changed: `src/lib/ai/report-pipeline/validator.ts` (new)
- **Learnings for future iterations:**
  - `compileMDX` from `next-mdx-remote/rsc` can be used headlessly (no components needed) for validation — just catches compilation errors
  - `sanitizeMdxProse` must be applied before compilation to escape bare `<` and `{` in AI-generated prose
  - The `KNOWN_COMPONENTS` set and `DATAREF_COMPONENTS` set must stay in sync with `src/components/mdx/components.tsx` — if new MDX components are added to the viewer, update both
  - Regex-based component parsing (`<PascalCase ...>`) is sufficient for validation — no need for a full AST parser since we control the component naming convention
  - The validator doesn't need React components registered in `compileMDX` for syntax validation — unknown components won't cause compilation errors in MDX (they're treated as custom elements)
