## Codebase Patterns
- Use `getDateStringInTimezone(trade.entryTime, userTimezone)` for date grouping
- Always group trades by ENTRY TIME, not exit time
- Frontend sends date strings via `toDateString()`, backend converts via `getDayBoundsInTimezone()`
- Timezone utilities are in `src/lib/shared/timezone.ts`
- User timezone fetched via `getUserTimezone(ctx.db, ctx.user.id)` helper

  - Journal date lookups used `toDateString(new Date(journal.date))` which converts to local time
  - In browsers behind UTC (e.g., PST), UTC midnight timestamps shift to previous day
  - Fix: Added `getUTCDateString()` to extract date directly from UTC
- **Key clarification:**
  - Calendar sidebar intentionally uses browser-local dates for grid generation
  - User clicks on "Jan 15" as shown → they want Jan 15's data
  - Backend handles timezone via `getDayBoundsInTimezone()`
  - This is correct behavior, not a bug
- **Learnings for future iterations:**
  - UTC midnight timestamps (like journal.date) represent calendar dates, not moments in time
  - Use `getUTCDateString()` for lookup maps from UTC midnight dates
  - Use `toDateString()` only for browser-local Date objects (calendar clicks)
  - The decision tree in TIMEZONE_SYSTEM.md is now comprehensive for all date handling cases
